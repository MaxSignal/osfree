61 * Description: This module implements a simple, generic, doubly linked list.
62 * Data objects of any type can be placed into a linked list
63 * created by this module. Futhermore, data objects of different
64 * types may be placed into the same linked list.
65 *
66 * Notes: This linked list implementation makes use of the concept of the
67 * current item. In any non-empty list, one item in the list will
68 * be designated as the current item. When any of the following
69 * functions are called, they will operate upon the current item
70 * only: GetItem, ReplaceItem, DeleteItem, GetTag, NextItem,
71 * PreviousItem, GetObject, ExtractItem, and ExtractObject. The
72 * user of this module may set the current item through the use of
73 * the GoToStartOfList, GoToEndOfList, NextItem, PreviousItem,
74 * and GoToSpecifiedItem functions.
75 *
76 * Since a linked list created by this module may contain items
77 * of different types, the user will need a way to identify items
78 * of different types which may be in the same list. To allow users
79 * to do this, the concept of an item tag is used. When an item is
80 * added to the list, the user must enter an item tag. The item
81 * tag is merely some identifier that the user wishes to associate
82 * with the item being placed into the list. When used as intended,
83 * each type of data item will have a unique tag associated with it.
84 * This way, all data items of the same type will have the same tag
85 * while data items of different types will have different tags.
86 * Thus, by using the GetTag function, the user can get the item
87 * tag for the current item without having to get the item from the
88 * list. This allows the user to differentiate between items of
89 * different types which reside in the same list.
90 *
91 * This module is single threaded. If used in a multi-threaded
92 * environment, the user must implement appropriate access controls.
93 *
94 * When an item is inserted or appended to a list, this module
95 * allocates memory on the heap to hold the item and then copies
96 * the item to the memory that it allocated. This allows local
97 * variables to be safely inserted or appended to a list. However,
98 * it should be noted that under certain circumstances a copy of the
99 * entire data item will NOT be made. Specifically, if the data item
100 * is a structure or array containing pointers, then the data pointed
101 * to by the pointers will NOT be copied even though the structure or
102 * array is! This results from the fact that, when an item is being
103 * inserted or appended to a list, the user provides just an address
104 * and size. This module assumes that the item to inserted or append
105 * lies in a contiguous block of memory at the address provided by the
106 * user. This module has no way of knowing the structure of the data
107 * at the specified address, and therefore can not know about any
108 * embedded pointers which may lie within that block of memory.
109 *
110 * This module now employs the concept of a handle. A handle is a
111 * reference to a specific item in a list which allows that item to
112 * be made the current item in the list quickly. Example: If you
113 * use the GetHandle function to get a handle for the current item
114 * (lets call the item B1), then, regardless of where you are in the
115 * list (or any reodering of the items in the list), you can make item
116 * B1 the current item by passing its handle to the GoToSpecifiedItem
117 * function. Alternatively, you could operate directly on B1 using
118 * the other handle based functions, such as GetItem_By_Handle, for
119 * example. GetItem_By_Handle gets the item associated with the
120 * specified handle without changing which item in the list is the
121 * current item in the list.
122 *
123 * The functions of this module refer to user data as either items or
124 * objects. The difference between the two is simple, yet subtle. It
125 * deals with who is responsible for the memory used to hold the data.
126 * In the case of an item, this module is responsible for the memory
127 * used to hold the user data. In the case of an object, the user
128 * is responsible for the memory used to hold the data.
129 *
130 * What this means is that, for functions adding ITEMS to a list,
131 * this module will be responsible for allocating memory to hold
132 * the user data and then copying the user data into the memory
133 * that was allocated. For functions which return items, this
134 * module will COPY the user data from the LIST into a buffer
135 * specified by the user. For functions which add objects to a
136 * list, the user provides a pointer to a block of memory holding
137 * user data. This block of memory was allocated by the user, and
138 * becomes the "property" of this module once it has been added to
139 * a LIST. For functions which return objects, a pointer to the
140 * memory where the data is stored is returned. As long as an item/object
141 * is in a LIST, this module will be responsible for the memory that
142 * is used to store the data associated with that item. This means that
143 * users of this module should not call free on an object returned by this
144 * module as long as that object is still within a list.
145 *
146 *****************************************************************************/
